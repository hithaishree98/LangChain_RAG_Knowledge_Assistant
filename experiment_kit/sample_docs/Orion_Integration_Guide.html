2. Retrieval Configuration

2.1 Default Settings

The retrieval subsystem uses the following defaults for Meridian's tenant. These were selected after the August 2024 tuning session with Mike Rodriguez's team:

retrieval:
  top_k: 2
  chunk_size: 1000
  chunk_overlap: 200
  embedding_model: nomic-ai/nomic-embed-text-v1.5
  hybrid_search: true
  reranker: disabled
  query_timeout_ms: 8000

The k=2 default was a deliberate choice after benchmarking against Meridian's question set — higher k values introduced more noise than signal for their document corpus, which leans heavily on short regulatory memos. Customers with longer-form documents typically benefit from k=4 or k=5.

2.2 Enabling the Reranker

The reranker is available on the Enterprise tier (included in Meridian's contract) but disabled by default to minimize latency. To enable:

orion-cli config set --tenant meridian --key retrieval.reranker --value cross-encoder

Expect a +180ms p95 latency impact and a measured +12% improvement in recall@5 based on the September benchmark.

2.3 Overriding k Per Query

The API supports per-query overrides via the retrieval_k parameter:

POST /v2/meridian/query
{
  "question": "What is the refund policy for delinquent accounts?",
  "retrieval_k": 5
}

3. Authentication and API Access

3.1 SSO Setup

SAML 2.0 SSO is configured with Meridian's Okta tenant. The following attributes are mapped on login:

  • NameID → Orion user identifier
  • email → primary contact
  • groups → Orion role (admin, analyst, viewer)

3.2 Service Accounts

Two service accounts are provisioned for machine-to-machine integration:

  • svc-meridian-etl — used by the nightly batch ingestion from the internal ledger
  • svc-meridian-sfdc — used by the Salesforce connector

Service account credentials rotate every 90 days. The next rotation is scheduled for 2024-11-18.

3.3 API Key Management

Every API request must include an X-API-Key header containing a valid API key generated from the Orion admin console under Settings > API Keys. Keys are scoped to the tenant and optionally restricted to read-only or read-write permission levels. Keys do not expire automatically but must be rotated quarterly per Orion's security policy and SOW Section 6.3. To rotate a key, follow these five steps in order: (1) generate a new key in the admin console under Settings > API Keys > Create Key, (2) update the new key in all client configurations and environment variables before revoking the old one, (3) verify connectivity by calling the /v2/health endpoint with the new key and confirming a 200 response, (4) revoke the old key from the admin console under Settings > API Keys > Revoke, (5) confirm the revocation event appears in the audit log under Audit > API Key Events. If an invalid or revoked key is presented, the API returns HTTP 401 with the error body {"error": "invalid_api_key"}.

3.4 Rate Limits

Each API call counts as one request regardless of payload size. Batch endpoints count as one request per item in the batch. Health check endpoints (/v2/health, /v2/ping) are exempt from rate limit accounting. The burst allowance permits short spikes above the per-minute limit and resets every 10 seconds.

When a client exceeds the rate limit, the API returns HTTP 429 with a Retry-After header indicating the number of seconds until the limit resets. The reset window is 60 seconds from the first request in the current window. Clients must implement exponential backoff starting at the value specified in the Retry-After header — do not retry immediately. Persistent rate limit violations from a single key are flagged for review and may result in temporary key suspension.

3.5 Session Management and JWT Tokens

The Orion admin console issues short-lived JWT tokens upon successful SSO authentication. Tokens expire after 30 minutes of inactivity. When a token expires, the user is automatically redirected to Meridian's Okta SSO endpoint for silent re-authentication. If silent re-authentication fails (for example, if the Okta session has also expired), the user is presented with the Okta login screen. Session tokens are signed with RS256 using a 2048-bit RSA key pair. The public key is available at https://admin.orion.io/.well-known/jwks.json for validation by Meridian's internal systems. Service account tokens do not expire on inactivity but are rotated on the 90-day schedule described in Section 3.2. Any token presented to the API after its expiry timestamp returns HTTP 401 with {"error": "token_expired"}.

4. Data Ingestion

4.1 Plaid Integration

Plaid transaction events flow via webhook to the Orion ingestion endpoint. The current Plaid schema version is v3.2.1. Plaid has announced deprecation of v3.2.1 in Q2 2025 in favor of v4.0. All tenants must complete migration before the deprecation date or event ingestion will fail silently.

Migrating from Plaid schema v3.2.1 to v4.0 requires the following five steps completed in order: (1) update the Plaid webhook destination URL in the Plaid developer dashboard to use the v4 endpoint (/v2/meridian/ingest/plaid/v4), (2) update the Orion event schema mapping in the admin console under Integrations > Plaid > Schema Version, (3) send a synthetic test event using the Plaid sandbox and confirm it appears in the Orion event log without errors, (4) monitor the plaid_schema_errors metric in the Orion Grafana dashboard for 24 hours to confirm zero schema validation failures, (5) notify your Orion FDE that migration is complete so they can disable the v3.2.1 compatibility shim on Orion's side.

Orion's v4.0 connector support is targeted for Q1 2025. Meridian has requested beta access — see TICK-4702.

4.2 Stripe Integration

Stripe payment events are pulled via the Stripe API using API version 2024-04-10. The integration runs on a 30-second polling interval with idempotency guaranteed via event ID.

4.3 Internal Ledger Batch

A nightly SFTP drop at 02:00 ET delivers the pipe-delimited ledger export. File naming convention: meridian_ledger_YYYYMMDD.pipe.gz. Typical file size: 2.4-3.8 GB compressed.

4.4 Webhook Delivery and Retry Policy

All Orion-sourced webhooks — including billing event notifications, ingestion status callbacks, and alert webhooks — use the following delivery and retry protocol. The initial delivery attempt is made within 10 seconds of the triggering event. If the receiving endpoint returns a non-2xx response or fails to respond within 30 seconds, Orion's webhook dispatcher marks the delivery as failed and schedules a retry. Retries follow an exponential backoff schedule: first retry at T+5 minutes, second retry at T+15 minutes, third retry at T+45 minutes, fourth retry at T+2 hours. After four failed retries the delivery is abandoned and a delivery-failure alert is posted to the #orion-alerts Slack channel. The triggering event is not lost — it is stored in the webhook event log accessible via the admin console under Integrations > Webhook Log and via the API at GET /v2/meridian/webhooks/events. Meridian can replay any failed webhook manually from this log for up to 30 days after the original trigger. Idempotency on the receiving end is Meridian's responsibility: Orion guarantees at-least-once delivery, not exactly-once. Meridian's receiving systems should key idempotency on the orion_event_id field present in every webhook payload.

4.5 Ingestion Monitoring

The ingestion pipeline exposes the following metrics in the Orion Grafana dashboard under the Meridian > Ingestion panel. Current event throughput (events per second, 1-minute average) is shown as ingest_events_per_second. The monthly cumulative ingestion counter is shown as ingest_monthly_total and updates every 15 minutes; Meridian's contracted limit of 200 million events per month is shown as a red threshold line. When the monthly counter reaches 85% of the contracted limit (170 million events), an automated alert is sent to the Meridian account owner (David Park) and to the primary technical contact (Mike Rodriguez). At 95% (190 million events), a second alert is sent with an estimated date of overage based on the trailing 7-day average daily ingestion rate. Overage events are billed at $0.08 per 10,000 events and appear as a line item on the following month's invoice. As of September 2024, Meridian is ingesting approximately 142 million events per month, leaving 29% headroom before overage applies.

5. Downstream Integrations

5.1 Salesforce Connector (v2.4.1)

Scored segments and customer 360 snapshots flow to the Salesforce Customer_360 object via the Orion-Salesforce connector, version 2.4.1.

Known limit: v2.4.1 caps bulk updates at 50,000 records/hour. Meridian's current volume is ~12,000/hour — well within spec, but trending upward. Monitor via the sfdc_bulk_throughput dashboard.

5.2 Snowflake Warehouse

Hourly export lands in the Snowflake external stage EXTERNAL_STAGE:ORION_EXPORT as Parquet files partitioned by hour. Retention: 30 days in the external stage, indefinite in Meridian's downstream curated zone.

5.3 Slack Alerts

P0 and P1 incidents post to #orion-alerts. P2+ incidents are summarized in a daily digest to the same channel at 09:00 ET.

5.4 Salesforce Connector Upgrade Path

The v3.x Salesforce connector, which removes the bulk update rate limit entirely by switching to Salesforce's Bulk API 2.0, is targeted for general availability in Q1 2025. The migration from v2.4.1 to v3.x is a configuration-only change and does not require a maintenance window. To upgrade: navigate to Integrations > Salesforce > Connector Version in the admin console, select v3.x, review the field mapping diff presented by the console (there are three field name changes in v3.x — see the migration guide in the admin console for the full list), confirm the mapping, and click Activate. The connector will switch to the new version on the next scheduled sync cycle (within 1 hour). Monitor the sfdc_bulk_throughput and sfdc_sync_errors dashboards for the first 24 hours after migration. If sync errors increase above the baseline, roll back via the same console by selecting v2.4.1 and clicking Activate. David Park recommends planning the v3.x migration alongside the planned seat expansion in Q1 2025 to ensure the connector can handle the increased data volume from additional analyst activity.

6. Operational Runbooks

6.1 If Login Latency Spikes

  1. Check the auth_p95_latency dashboard (Grafana > Orion > Meridian > Auth).
  2. If p95 > 1500ms for more than 5 minutes, open a P1 ticket.
  3. Workaround for analysts: clear browser cache, use incognito mode.
  4. Historical reference: TICK-4521 (load balancer sticky-session issue, Sept 2024).

6.2 If Snowflake Export Is Delayed

  1. Check the hourly export dashboard. Expected lag: under 8 minutes.
  2. Lag over 30 minutes: check the Snowflake warehouse status.
  3. Lag over 60 minutes: escalate to the Orion FDE (David Park).

6.3 Emergency Rollback

If a configuration change causes widespread issues, roll back via:

orion-cli config rollback --tenant meridian --to-revision previous

6.4 Scheduled Maintenance Window Procedure

Orion's standard maintenance window for Meridian's tenant is Sunday 02:00-04:00 UTC (Saturday evening US time). During this window, the following may be performed without customer notification: configuration parameter changes, index rebuilds for newly ingested documents, certificate rotations, and non-breaking dependency updates. Changes that require API downtime or that modify the data schema must be communicated to Mike Rodriguez and David Park at least 5 business days in advance via email and must be confirmed in the change management ticket system. Emergency changes outside the maintenance window require P0 or P1 ticket classification, documented approval from the on-call VP Engineering, and notification to Meridian's primary contact within 30 minutes of the change start. Post-maintenance validation steps: (1) run GET /v2/health and confirm 200; (2) verify auth p95 latency is below 1,000ms on the Grafana dashboard; (3) submit a test query from the admin console and confirm a valid response; (4) check the Snowflake export lag dashboard and confirm the most recent export is within 8 minutes of schedule; (5) post a maintenance completion notice to #orion-alerts in Slack with the change summary and validation results.

6.5 Latency SLA Breach Playbook

If any Orion endpoint exceeds its SLA latency threshold for more than 5 consecutive minutes, the following steps should be executed in order. First, check the Grafana SLA dashboard (Orion > Meridian > SLA Overview) to confirm the breach is real and not a monitoring artifact — spikes shorter than 5 minutes do not trigger the SLA credit calculation. Second, identify which endpoint is breaching: auth p95 threshold is 1,000ms, query p95 threshold is 5,000ms, ingestion p95 threshold is 3,000ms. Third, open a P1 ticket immediately with the affected endpoint, breach duration, and observed latency. The on-call engineer must acknowledge within 2 hours per SLA Section 2.2. Fourth, notify the Meridian primary contact (Mike Rodriguez) by email and Slack DM with the ticket number and an estimated resolution time. Fifth, if the breach persists past 4 hours without a resolution ETA from the on-call engineer, escalate to the Orion team lead. If unresolved at 8 hours, the ticket auto-escalates to VP Engineering. At 24 hours the account executive (Elena Martinez) is engaged to manage the customer relationship. Credits accrue per SOW Section 2.3 and are applied to the following invoice automatically — no manual credit request is required from Meridian.

6.6 Audit Log Review Procedure

Meridian's audit log is accessible to users with the admin-role permission via the admin console under Audit > Event Log, or programmatically via GET /v2/audit with an admin-scoped API key. The log retains all events for 7 years in immutable storage and cannot be modified or deleted. For compliance reviews, export the relevant date range as CSV from the admin console (Audit > Export > CSV) or as JSON via the API. When conducting a scheduled quarterly compliance review, the recommended procedure is: (1) export the previous quarter's events filtered by event_type=admin_action and event_type=data_export; (2) cross-reference with the change management ticket system to confirm all admin actions have corresponding approved tickets; (3) verify no data export events occurred outside of approved export windows; (4) confirm all API key creation events have a matching ticket in the key rotation log; (5) submit the export file to Meridian's compliance team with the audit period, event count, and any anomalies noted. Anomalies should be reported to David Park within 24 hours of discovery.

7. Glossary

FDE
Forward Deployed Engineer — the Orion engineer assigned to Meridian.
QBR
Quarterly Business Review.
RPO
Recovery Point Objective — maximum acceptable data loss window.
RTO
Recovery Time Objective — maximum acceptable downtime for recovery.
Sticky session
Load balancer feature that pins a user to a specific backend node for their session duration.
BM25
Best Match 25 — a probabilistic lexical ranking algorithm used alongside dense embeddings in Orion's hybrid search.
RRF
Reciprocal Rank Fusion — the merging algorithm that combines dense and BM25 result sets into a single ranked list.
Cross-encoder reranker
A second-stage retrieval model that scores each candidate chunk against the query jointly, improving precision at the cost of additional latency.
Parent-child indexing
A retrieval strategy where small child chunks are searched for embedding similarity but their larger parent chunks are returned as context, balancing search precision with answer completeness.

8.2 Ingestion Failures

If documents uploaded via POST /v2/meridian/ingest do not appear in the document list within 5 minutes, check the ingestion status endpoint at GET /v2/meridian/ingest/{job_id}/status. Common failure reasons include: unsupported file format (supported: PDF, DOCX, HTML, TXT, JSON), file size exceeding 50 MB, malformed PDF (no extractable text layer — scanned PDFs require OCR pre-processing before ingestion), and character encoding errors in plain-text files (UTF-8 is required). If the status endpoint returns {"status": "failed", "reason": "extraction_error"}, the file content could not be extracted. For scanned PDFs, use orion-cli preprocess --ocr input.pdf output.pdf before re-ingesting.

8.3 Snowflake Export Troubleshooting

If the Snowflake external stage is not receiving new Parquet files on the expected hourly schedule, verify the following in order. First, check that the Snowflake warehouse ORION_WH is running and not suspended — a suspended warehouse causes the COPY INTO command to queue indefinitely. Second, verify the external stage credentials have not expired: the storage integration credentials rotate every 180 days, and the next rotation for Meridian's stage is 2025-01-15. Third, confirm that the Orion export job is running by checking GET /v2/meridian/exports/status — the last_export_at timestamp should be within the past 90 minutes. If last_export_at is older than 90 minutes, the export job has stalled and David Park should be paged. Fourth, if files are arriving in the stage but not visible in Meridian's curated zone, the issue is downstream of Orion and should be investigated by Meridian's Snowflake team. David Park can provide the raw Parquet file list in the external stage on request.
